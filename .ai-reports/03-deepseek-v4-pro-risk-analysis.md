## 2. Auth / Session Risks

### 2.1 Dual Auth Path Ambiguity (HIGH)

**File**: [`backend/app/middleware/auth_middleware.py`](backend/app/middleware/auth_middleware.py:58)

```python
user_result = await self._try_user_key(request, api_key, device_id, path)
if user_result is not None:
    return user_result  # None means "try legacy", otherwise a response
```

The fallback pattern "try user-key first, then legacy" means:

- A **malformed or expired user-key** silently falls through to legacy validation (line 93: `return None` on `not record`). If a key exists in both systems, ambiguous which one rejected.
- Error codes differ between paths. User-key returns `"blocked_user"` / `"inactive_user"` / `"payment_pending"`, but legacy only returns `"invalid_key"` / `"device_mismatch"`.
- The client cannot distinguish which auth system rejected them, making debugging difficult.

### 2.2 Deterministic Admin Session Cookie (MEDIUM)

**File**: [`backend/app/api/admin_routes/utils.py`](backend/app/api/admin_routes/utils.py:43)

```python
def _admin_session_cookie(request: Request) -> str:
    auth_secret = f"{settings.auth.admin_username}:{settings.auth.admin_password}"
    raw = f"{settings.auth.hash_salt}:{auth_secret}".encode("utf-8")
    return hashlib.sha256(raw).hexdigest()
```

The cookie value is deterministic — same credentials always produce the same token. There is no session ID, no nonce, no expiration embedded in the cookie itself (expiry is only cookie `max_age=12h`). If an attacker obtains the cookie, they can reuse it until password changes.

### 2.3 Master Key Required for Write Operations (LOW)

**File**: [`backend/app/api/routes.py`](backend/app/api/routes.py:62)

```python
def _ensure_master_key(request: Request) -> None:
    key_record = request.state.api_key_record
    if not key_record or key_record.get("key_type") != "master":
        raise HTTPException(403, "Master key required...")
```

The master key is created once via `api_keys.ensure_master_key()` (database.py:440). If the master key is lost or the DB is wiped, there is no recovery mechanism — no admin can create new keys.

### 2.4 Admin Token for Key Management (LOW)

**File**: [`backend/app/middleware/auth_middleware.py`](backend/app/middleware/auth_middleware.py:48)

Uses `hmac.compare_digest` for `x-admin-token` header comparison — correct timing-safe comparison. However, the token is passed as a plain HTTP header, so it must only be used over HTTPS.

## 3. Database / Data Consistency Risks

### 3.1 Dual Database Systems — Schema Drift (CRITICAL)

**Evidence**: Two independent DB layers coexist:

| Layer | File | Tables | Driver |
|-------|------|--------|--------|
| Legacy raw SQL | [`backend/app/core/database.py`](backend/app/core/database.py:1) | `api_keys`, `usage_events`, `model_registry`, `field_mappings`, `autofill_proposals`, `failed_payloads`, `settings`, `domain_locators`, `exam_attempts`, `exam_learned`, `exam_config`, `sign_hashes`, `training_jobs`, `backups`, `captcha_proposals`, `automation_methods` (17 tables) | `sqlite3` module |
| New ORM | [`backend/app/core/db.py`](backend/app/core/db.py:1) + [`backend/app/core/models.py`](backend/app/core/models.py:1) | `users`, `subscription_plans`, `user_subscriptions`, `payment_records`, `user_api_keys`, `user_api_key_devices`, `usage_cycles`, `audit_logs`, `backup_runs` (9 models) | SQLAlchemy |

**Overlapping concerns**:
- `api_keys` (raw SQL) vs `user_api_keys` (SQLAlchemy) — both represent API keys but have different schemas. Auth middleware tries user_keys first, then falls back to api_keys.
- `usage_events` (raw SQL) vs `usage_cycles` (SQLAlchemy) — duplicate usage tracking.
- `platform_settings` (raw SQL) — accessed by BOTH systems. Telegram bot uses raw `text()` SQL against it (telegram_bot.py:1306-1308) while services read via `Database.get_setting()`.

**Risk**: If a column rename or schema change happens in one system but not the other, inconsistent data across the two representations. The inline migrations in `database.py:init()` (lines 352-421) modify raw SQL tables; SQLAlchemy tables are managed by Alembic. No synchronization between the two.

### 3.2 Inline DDL + Migrations with No Versioning (CRITICAL)

**File**: [`backend/app/core/database.py`](backend/app/core/database.py:100)

- 250+ lines of `CREATE TABLE IF NOT EXISTS` (lines 100-351)
- 70+ lines of `PRAGMA table_info` + `ALTER TABLE` (lines 352-421)
- **No migration history table, no versioning, no rollback capability**
- Runs on every startup — idempotent but wastes time
- If a column rename fails mid-execution (e.g., lines 376-385: `ALTER TABLE model_registry RENAME COLUMN model_name TO ai_model_name`), the database enters an unknown state
- SQLite's `ALTER TABLE` is limited (no `DROP COLUMN` in older versions, restrictive rename)

### 3.3 File-Based State with No Locking (HIGH)

**Telegram bot states**: [`backend/app/services/telegram_bot.py`](backend/app/services/telegram_bot.py:85)

```python
self._state_file = _data_dir / "telegram_user_states.json"
```

- User conversation states persisted to JSON file
- No file locking — if two bot processes write simultaneously, data corrupts
- Only 30-min TTL filter on load (line 102), no cleanup of stale entries on write
- `_save_states()` (line 108) writes entire dictionary — if the file is large, this is inefficient and can be interrupted mid-write

### 3.4 Question/Learned Data in JSON Files (MEDIUM)

**Files**: [`data/questions/questions.json`](data/questions/questions.json:1) (146KB), [`data/questions/questions_learned.json`](data/questions/questions_learned.json:1)

- Question database is a flat JSON file — no indexing, loaded entirely into memory
- `questions_learned.json` is exported from `exam_learned` table periodically but no guaranteed consistency with the DB
- Atomic writes use temp-file + rename pattern (`_write_json_atomic` at routes.py:112), which is safe on local filesystem but unsafe on NFS

### 3.5 `create_all_tables()` Called at Startup (MEDIUM)

**File**: [`backend/app/core/container.py`](backend/app/core/container.py:70)

```python
from app.core.db import create_all_tables
create_all_tables()
```

SQLAlchemy's `Base.metadata.create_all()` is called unconditionally at startup. In production with Alembic migrations, this is redundant but harmless with `IF NOT EXISTS` semantics. However, if a migration added a column that `create_all()` doesn't know about (because the model was updated after migration ran), it will try to recreate and could fail.

## 4. API Validation Risks

### 4.1 Large Payload Handling (MEDIUM)

**File**: [`backend/app/api/routes.py`](backend/app/api/routes.py:50)

```python
_REPORT_MAX_BYTES = 5 * 1024 * 1024  # 5 MB
```

- Report endpoint has size limit (line 538), but `/v1/solve` and `/v1/exam/solve` accept base64 images with **no size limits** before decoding
- Pydantic `min_length=1` on `payload_base64` (schemas.py:19) only checks string is non-empty, not its decoded size
- An attacker could send a 100MB base64 string and exhaust memory during `b64decode`

### 4.2 Missing Input Sanitization on File Paths (HIGH)

**File**: [`backend/app/api/routes.py`](backend/app/api/routes.py:376)

```python
path = (scripts_dir / file_name).resolve()
if scripts_dir.resolve() not in path.parents or not path.is_file():
    raise ValueError(f"Invalid userscript file: {file_name}")
```

The userscript sync endpoint validates that resolved path stays within `scripts_dir` parents — this is correct. However, the `_read_automation_script()` function (line 862) only checks `step_id in _AUTOMATION_SCRIPT_IDS` and does `script_path.read_text()` — also safe due to the allowlist.

### 4.3 Domain Validation Basic (LOW)

**File**: [`backend/app/core/database.py`](backend/app/core/database.py:44)

```python
@staticmethod
def _normalize_domain(domain: str | None) -> str:
```

Domain normalization strips protocol, path, and port. But no validation against a list of known domains — any string that looks like a domain passes. This is used in rate-limit buckets and allowed-domain checks, so invalid domains create useless buckets.

### 4.4 Rate Limiting — In-Memory Only (MEDIUM)

**Files**:
- [`backend/app/api/routes.py`](backend/app/api/routes.py:57) — `_report_buckets: dict` — global mutable state for report rate limiting
- [`backend/app/middleware/rate_limit_middleware.py`](backend/app/middleware/rate_limit_middleware.py:26) — `self._events: dict` — per-key rate limiting

Both are:
- **Lost on restart** — an attacker can restart and bypass limits
- **Per-process only** — with `--workers 2` in Dockerfile:57, each worker has its own buckets, effectively doubling the allowed rate
- **No persistence** — cannot enforce limits across deployments

## 5. Error Handling Gaps

### 5.1 Generic 500 Errors with Hidden Details (MEDIUM)

**File**: [`backend/app/api/routes.py`](backend/app/api/routes.py:498)

```python
except Exception as error:
    container.usage_service.record(..., status="error")

```

This pattern repeats 6+ times in routes.py. The actual error message is hidden. Clients only see `"captcha solve failed"`. In production with `debug=False`, the traceback is lost. Logging happens, but there is no structured error code to correlate client errors with server logs.

### 5.2 Silently Degraded Tesseract (HIGH)

**File**: [`backend/app/services/exam_service.py`](backend/app/services/exam_service.py:32-63)

```python
try:
    import pytesseract
    _TESSERACT_AVAILABLE, _TESSERACT_CMD = _configure_tesseract()
except ImportError:
    _TESSERACT_AVAILABLE = False
```

Tesseract is probed at module import time. If the binary is not found, `_TESSERACT_AVAILABLE = False` and all OCR calls return empty strings via `_ocr_text()` (not shown but inferred). **No runtime error, no alert** — the exam solver silently falls back to hash matching, then LLM, then returns `"none"`. This can go unnoticed for days while users see degraded accuracy.

### 5.3 Telegram Bot States Lost on Crash (MEDIUM)

## 6. Background Jobs, Queues, Cron, or Async Side Effects

### 6.1 BackgroundTasks — Fire-and-Forget (MEDIUM)

**File**: [`backend/app/api/routes.py`](backend/app/api/routes.py:730)

```python
background_tasks.add_task(_save_exam_offline_dataset_safe, ...)
background_tasks.add_task(_export_learned_to_json_safe, container)
```

FastAPI's `BackgroundTasks` run after the response is sent. If the server crashes mid-task, data is lost. No retry, no queue persistence. The offline dataset saving (creating directories, writing images, updating index JSON) has no transactional guarantees with the DB write that triggers it.

### 6.2 Solver Service Thread Pool (LOW)

**File**: [`backend/app/core/container.py`](backend/app/core/container.py:76)

```python
solver = SolverService(workers=settings.queue.workers, max_pending_jobs=500, ...)
```

The solver service uses a thread pool with a max queue of 500. If the queue fills, new submissions are rejected. No backpressure mechanism — the client gets a 500 error.

### 6.3 OCR Thread Pool Per ExamService (LOW)

**File**: [`backend/app/services/exam_service.py`](backend/app/services/exam_service.py:763)

```python
q_future = self._ocr_pool.submit(self._ocr_text, q_img)
```

The exam service creates its own thread pool for parallel OCR of question + 4 options. This pool's size is controlled by `ocr_concurrency` config (default 2). With many concurrent exam solve requests, thread pool exhaustion can cause queuing.

### 6.4 No Cron/Scheduled Jobs (INFO)

There are no scheduled jobs (cron, Celery beat, APScheduler) in the codebase. Backup rotation happens inline during `full_backup()` calls (backup_service.py:77). Stale state cleanup happens opportunistically (telegram bot: 30-min filter on load, rate limit: periodic prune on requests).

### 6.5 Telegram Bot as Separate Process (LOW)

**File**: [`docker-compose.yml`](docker-compose.yml:39)

The Telegram bot runs as a separate Docker container (`sa-helper-telegram-bot`) with the same image but different command. It shares the SQLite database via Docker volume. Two processes writing to the same SQLite file — while SQLite WAL mode handles this, there is no coordination for schema migrations between the two containers.

## 10. Safe Migration Order

If migrating to a new architecture (e.g., modular monolith recommended in 02-qwen report), follow this order to minimize risk:

1. **Add tests for auth middleware** — ensure dual-path behavior is covered before refactoring
2. **Extract business logic from [`routes.py`](backend/app/api/routes.py:1)** into dedicated service methods first (lowest risk, highest impact)
   - Move `_save_exam_offline_dataset()` → `exam_service.save_feedback_dataset()`
   - Move `_userscript_*` functions → a new `UserscriptAccessService`
   - Move `_compose_stall_flow_payload()` → `AutomationService`
3. **Replace inline DDL migrations** in [`database.py`](backend/app/core/database.py:100) with Alembic migrations for legacy tables
4. **Consolidate the dual DB layers** — choose one (SQLAlchemy recommended) and migrate legacy tables one at a time:
   - First: `platform_settings` (most shared)
   - Then: `api_keys` → merge with `user_api_keys`
   - Then: `usage_events` → merge with `usage_cycles`
5. **Extract the Telegram bot** into a separate module with clear boundary (it's already a separate process)
6. **Introduce repository interfaces** — hide raw SQL behind abstract repositories so services don't care about DB implementation
7. **Add proper error codes** — replace generic 500s with typed error responses
8. **Replace in-memory rate limiting** with Redis (already has Redis config scaffolding at config.py:162-165)
9. **Split frontend `App.jsx`** into context providers and route-level components
10. **Extract extension modules** into independently testable units

## 12. Recommended Rollback Strategy

### Pre-Deployment Checklist

1. **Take a full backup** before any deployment: `POST /admin/api/backups/full` (via [`backup_service.py`](backend/app/services/backup_service.py:30))
2. **Verify backup integrity**: check `backup_*/database.sqlite` exists and is valid SQLite
3. **Tag the current Docker image** before pushing a new one
4. **Document the current `alembic` head** revision

### Rollback by Severity

| Scenario | Rollback Action | Downtime |
|----------|----------------|----------|
| **DB migration fails** | Restore from backup via [`backup_service.restore_from_backup()`](backend/app/services/backup_service.py:166), restart server | 5-15 min |
| **Auth middleware broken** (all users locked out) | Revert to previous Docker image tag, restart containers | 2-5 min |
| **Exam solver degraded** (wrong answers) | Roll back `exam_service.py` changes or flip `exam.learning_enabled=false` in platform_settings | 0 min (runtime config) |
| **Telegram bot broken** | Stop `telegram-bot` container, fix code, restart. Users can still use extension directly | 0 min for API |
| **Rate limiting too aggressive** | Increase `RATE_LIMIT_RPM` env var or set per-key override via admin dashboard | 0 min (runtime config) |
| **Extension broken** (bad manifest/background.js) | Republish previous extension version. Users must manually update from admin dashboard | Variable (user-dependent) |

### Critical Safety Switches (Runtime Toggles)

These can be toggled without deployment via the `platform_settings` table (admin dashboard):

| Setting Key | Effect When Disabled |
|-------------|---------------------|
| `exam.learning_enabled` | Stops all self-learning — safe mode for exam solver |
| `alerts.whatsapp_enabled` | Stops WhatsApp notifications |
| `telegram.bot_token` (empty) | Disables Telegram bot |
| `automation.dynamic_methods_enabled` | Falls back to file-based automation scripts |

### Database Rollback Specifics

The inline migrations in [`database.py:init()`](backend/app/core/database.py:352) have NO undo capability. If an `ALTER TABLE ADD COLUMN` succeeds but causes issues:

1. Manual SQL: `ALTER TABLE xxx DROP COLUMN yyy` (SQLite 3.35+)
2. Or: restore from pre-migration backup
3. Recommended: migrate ALL legacy table DDL to Alembic (which supports `downgrade()`)
