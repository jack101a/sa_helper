## 1. Current Architecture Style

**Monolithic full-stack monorepo** with three coupled tiers:

| Tier | Technology | Style |
|------|-----------|-------|
| Backend | Python 3.11 / FastAPI | Script-style entrypoint + service layer + raw-SQL facade |
| Frontend | React 18 / Vite | SPA with React Query, no state manager, prop-drilling from `App.jsx` |
| Extension | Chrome MV3 | Content scripts + service worker, no bundler |

The backend follows a **layered architecture** (routes → services → database) but with significant deviations:
- `routes.py` contains business logic, not just routing
- `database.py` is a **God Facade** — 560 lines, 80+ proxy methods, inline DDL, inline migrations
- Services directly import from `database.py` and `db.py` (two different DB layers)
- No clear separation between domain logic and infrastructure

**Key pattern**: "Container" DI (`backend/app/core/container.py`) wires all services at startup, but services reach across layers freely.

## 3. Current Module Boundaries

The intended boundaries are:

```
routes.py (HTTP layer)
    ↓
services/*.py (business logic)
    ↓
core/database.py (legacy SQLite facade)
    ↓
core/repositories/*.py (raw SQL per domain)

core/db.py (SQLAlchemy engine)
    ↓
core/models.py (ORM models)
    ↓
services/*.py (new domain services)
```

**Actual boundaries are porous:**

- `routes.py` imports directly from `core/database.py` (line 19), `core/security.py` (line 20), `core/userscript_utils.py` (line 21), `core/automation_method_utils.py` (line 22) — bypassing services
- `routes.py` contains 200+ lines of helper functions (`_userscript_*`, `_save_exam_offline_dataset`, `_compose_stall_flow_payload`) that are business logic, not routing
- `analytics.py` (admin route) calls `container.solver_service.submit()` directly (line 45-53) — a route reaching into a service for OCR processing
- `telegram_bot.py` imports `UserKeyService` and `get_settings()` directly (lines 538-539) — a service reaching into another service + config
- `backup_service.py` imports `get_session` and `BackupRun` model directly (lines 16, 133) — mixing raw SQLite and SQLAlchemy in one service
- `auth_middleware.py` imports `UserKeyService` and `get_session` inside a method (lines 83-84) — lazy import to avoid circular dependency

## 5. Coupling Problems

### 5.1 Dual Database Systems (Highest Risk)

**Evidence**:
- `backend/app/core/database.py` — raw SQLite with 17 tables
- `backend/app/core/db.py` — SQLAlchemy with 9 models
- `backend/app/core/models.py` — SQLAlchemy ORM models
- `backend/app/core/container.py` lines 62-71 — initializes BOTH systems

**Problem**: Two independent database layers coexist with overlapping concerns:
- `api_keys` table (raw SQL) vs `user_api_keys` table (SQLAlchemy)
- `usage_events` table (raw SQL) vs `usage_cycles` table (SQLAlchemy)
- `platform_settings` table (raw SQL) — accessed by BOTH systems
- `backups` table (raw SQL) vs `backup_runs` table (SQLAlchemy)

The auth middleware (`auth_middleware.py`) tries user-key first, then falls back to legacy keys — this is runtime coupling that makes debugging difficult.

### 5.2 Circular Import Risk

**Evidence**:
- `auth_middleware.py` line 83: `from app.services.user_key_service import UserKeyService` — lazy import inside method
- `telegram_bot.py` line 538: `from app.services.user_key_service import UserKeyService` — lazy import
- `telegram_bot.py` line 539: `from app.core.config import get_settings` — direct config access
- `analytics.py` line 84: `from app.core.db import get_session` — direct DB access in route
- `backup_service.py` line 16: `from app.core.db import get_session` — direct DB access in service

These lazy imports indicate circular dependency problems that are being worked around rather than solved.

### 5.3 Service-to-Service Coupling

- `telegram_bot.py` creates its own `UserKeyService` instance (line 539) instead of receiving it via DI
- `backup_service.py` directly queries SQLAlchemy models (line 133: `from app.core.models import BackupRun`)
- `exam_service.py` receives `Database` (legacy) but also does its own file I/O
- `solver_service.py` receives `ModelRouter` which receives `Database` — deep dependency chain

### 5.4 Frontend Prop-Drilling

**Evidence**: `frontend/src/app/App.jsx` (220 lines)

All state lives in `App.jsx`:
- 15+ `useState` declarations (lines 102-117)
- 4 handler hooks receiving 20+ props each
- `DashboardPage` receives 14 props (lines 33-39)
- `SettingsPanel` receives 12+ props (line 212)
- `ModelsPanel` + `MappingsPanel` share editing state via lifted state

No context providers (except theme), no state manager. Adding a new panel requires modifying `App.jsx`.

### 5.5 Frontend-Backend Tight Coupling

**Evidence**: `frontend/src/app/hooks/useAdminData.js`

The `fetchBootstrap` call returns a single massive JSON payload with everything:
- usage stats, api_keys, allowed_domains, model_registry, field_mappings, datasets_files, master_key_info

This means:
- Every panel refreshes all data on every 30s interval
- Adding a new data type requires changing the bootstrap endpoint
- No independent caching per resource

## 7. Fragile Areas

### 7.1 `database.py` Inline DDL + Migrations (Critical)

**File**: `backend/app/core/database.py`, `init()` method (lines 100-421)

- 250+ lines of `CREATE TABLE IF NOT EXISTS` SQL
- 70+ lines of `PRAGMA table_info` + `ALTER TABLE` checks
- No versioning, no rollback, no migration history
- If a column rename fails mid-execution, the database is in an unknown state
- The `init()` method runs on every startup — it's idempotent but slow

### 7.2 In-Memory Report Rate Limiting

**File**: `backend/app/api/routes.py`, lines 57-109

```python
_report_buckets: dict[tuple[int, str], deque[float]] = {}
```

Global mutable state. Lost on restart. No persistence. Will grow unbounded if prune interval is missed.

### 7.3 Telegram Bot State in JSON File

**File**: `backend/app/services/telegram_bot.py`, lines 83-114

User conversation states stored in `data/telegram_user_states.json`. No locking, no atomicity, no TTL cleanup (only 30-min filter on load). If two bot processes write simultaneously, data corrupts.

### 7.4 Dual Auth Path in Middleware

**File**: `backend/app/middleware/auth_middleware.py`, lines 58-78

The fallback pattern ("try user-key, if None try legacy") means:
- A malformed user-key silently falls through to legacy validation
- Error messages are ambiguous — client can't tell which system rejected them
- Device binding logic differs between the two paths

### 7.5 Extension Content Script Injection Order

**File**: `extension/manifest.json`, lines 38-111

15 content scripts injected across 6 content script declarations with different `run_at` timings and `world` contexts. The order matters but is implicit in the manifest array. No build system validates dependencies.

### 7.6 `exam_service.py` Module-Level Tesseract Detection

**File**: `backend/app/services/exam_service.py`, lines 32-63

Tesseract is probed at import time. If the binary is not found, the module logs a warning but continues. Any code that later calls OCR will silently return empty strings.

## 9. Suggested Target Architecture Options

### Option A: Clean Layered Architecture (Incremental Refactor)

```
backend/
  app/
    api/
      v1/              # Thin route handlers — validation + delegate to services
      admin/           # Same pattern
    services/          # Business logic only — no DB, no HTTP, no file I/O
    domain/            # Entities, value objects, domain rules
    infrastructure/
      database/        # Repository implementations (both SQLite and SQLAlchemy)
      ai/              # ONNX, Tesseract, LiteLLM clients
      storage/         # File I/O, backup, extension packaging
      messaging/       # Telegram bot, WhatsApp alerts
    core/
      config.py
      container.py     # DI wiring
      middleware/
```

**Approach**: Extract business logic from routes into services. Extract DB access from services into repositories. Keep the dual-DB system but isolate it behind a unified repository interface.

### Option B: Domain-Driven Design (Vertical Slices)

```
backend/
  app/
    captcha/           # Captcha domain
      api/
      service/
      repository/
      models/
    exam/              # Exam domain
      api/
      service/
      repository/
      models/
      hashing/
      ocr/
    autofill/          # Autofill domain
      api/
      service/
      repository/
    users/             # User/subscription domain
      api/
      service/
      repository/
      models/          # SQLAlchemy ORM
    shared/            # Cross-cutting
      auth/
      config/
      database/
      middleware/
```

**Approach**: Group by business capability, not by technical layer. Each domain owns its API, service, and data access. Shared infrastructure in `shared/`.

### Option C: Microservices (Long-term)

```
captcha-service/       # ONNX OCR only
exam-service/          # Hash → OCR → LLM pipeline
autofill-service/      # Rule engine
user-service/          # Users, subscriptions, payments, keys
admin-service/         # Dashboard API
gateway/               # API gateway (FastAPI or Nginx)
```

**Approach**: Split into independently deployable services. Requires message queue, shared config, distributed tracing.

### Option D: Modular Monolith (Recommended Middle Ground)

```
backend/
  app/
    modules/
      captcha/
        __init__.py    # Module registration
        routes.py
        service.py
        repository.py
      exam/
        __init__.py
        routes.py
        service.py
        repository.py
        hashing.py
      autofill/
        __init__.py
        routes.py
        service.py
        repository.py
      users/
        __init__.py
        routes.py
        service.py
        repository.py
        models.py      # SQLAlchemy
      admin/
        __init__.py
        routes/        # 16 admin route files
    shared/
      database/        # Unified DB abstraction
      auth/
      config/
      middleware/
    main.py            # Module discovery + wiring
```

**Approach**: Keep single deployable unit but enforce module boundaries. Each module has its own routes, service, and repository. Shared concerns in `shared/`. Modules communicate via interfaces, not direct imports.

## 10. Tradeoffs of Each Option

| Option | Pros | Cons | Effort | Risk |
|--------|------|------|--------|------|
| **A: Clean Layered** | Familiar pattern, incremental, low risk | Does not solve domain coupling, routes.py still large | Medium (2-4 weeks) | Low |
| **B: DDD Vertical Slices** | Clear domain boundaries, easy to test, scales to microservices later | Requires understanding all domain boundaries upfront, significant restructuring | High (4-8 weeks) | Medium |
| **C: Microservices** | Independent scaling, team autonomy, technology flexibility per service | Operational complexity, distributed debugging, network latency, data consistency | Very High (3-6 months) | High |
| **D: Modular Monolith** | Best of A+B — clear boundaries without operational overhead, easy to extract services later | Requires discipline to maintain boundaries, initial restructuring effort | Medium-High (3-6 weeks) | Medium-Low |

**Recommendation**: **Option D (Modular Monolith)** as the target, with **Option A (Clean Layered)** as the first migration step.

Rationale:
- The project is a single-team, single-deployment system — microservices are overkill
- Domain boundaries are already somewhat clear (captcha, exam, autofill, users)
- A modular monolith can later be split into microservices if needed
- The incremental approach (A → D) reduces risk

## 11. Recommended Architecture Direction

### Phase 1: Stabilize (Week 1-2)
1. **Extract business logic from `routes.py`** into dedicated service methods
   - Move `_save_exam_offline_dataset` → `exam_service.save_feedback_dataset()`
   - Move `_compose_stall_flow_payload` → `automation_service.compose_stall_flow()`
   - Move userscript access control → `userscript_service.check_access()`
   - Move report rate limiting → `rate_limiter.check_report_limit()`

2. **Stop the `database.py` proxy pattern**
   - Callers should use `db.api_keys.get_api_key_by_hash()` not `db.get_api_key_by_hash()`
   - Remove the 80+ proxy methods

3. **Consolidate duplicate endpoints** in `analytics.py`
   - Remove `_legacy/*` routes or redirect them

### Phase 2: Modularize (Week 3-5)
4. **Create module boundaries** for captcha, exam, autofill, users
   - Each module gets its own `routes.py`, `service.py`, `repository.py`
   - Move related files from `services/`, `repositories/`, `api/` into modules

5. **Unify database access** behind a single abstraction
   - Either migrate all legacy tables to SQLAlchemy OR
   - Create a `Repository` base class that both SQLite and SQLAlchemy implementations conform to

6. **Split `telegram_bot.py`** into:
   - `telegram_bot.py` — bot framework + command routing
   - `telegram_state.py` — state machine
   - `telegram_payments.py` — payment flow
   - `telegram_users.py` — user queries

### Phase 3: Frontend Restructure (Week 5-6)
7. **Introduce React Context or Zustand** for shared state
   - Remove prop-drilling from `App.jsx`
   - Each panel manages its own data fetching

8. **Split `SettingsPanel.jsx`** (679 lines) into sub-components
   - `ApiKeySettings`, `DomainSettings`, `RateLimitSettings`, `CloudBackupSettings`

9. **Decouple bootstrap endpoint** — use individual React Query endpoints per resource

### Phase 4: Extension (Week 6-8)
10. **Add build system** (Vite or esbuild) for the extension
    - Proper module bundling instead of manifest script ordering
    - Tree shaking, minification, source maps

11. **Split `background.js`** (1711 lines) into modules
    - Message router, state manager, API client, notification handler

## 12. Questions Claude Opus Should Answer

1. **Dual database strategy**: Should the legacy raw-SQL tables (`api_keys`, `usage_events`, `model_registry`, `field_mappings`, etc.) be migrated to SQLAlchemy, or should both systems coexist long-term? What is the migration risk?

2. **Telegram bot process model**: The bot runs inside the FastAPI process via `START_TELEGRAM_BOT_IN_API` env var. Should it be a separate process? What are the implications for state sharing and notifications?

3. **Exam service hash functions**: `_djb2_hash`, `_phash`, `_dct_2d` are module-level functions in `exam_service.py` and imported directly by `routes.py`. Should these be in a shared `hashing` module, or should the route call the service instead?

4. **`database.py` proxy pattern**: The 80+ proxy methods exist for backward compatibility. What is the safest way to deprecate them — gradual removal, or keep them forever?

5. **Frontend bootstrap endpoint**: The `/admin/api/bootstrap` endpoint returns everything in one call. Should this be split into per-resource endpoints, or is the single-call approach acceptable for the current scale?

6. **Extension build system**: The extension has no bundler — scripts are loaded in manifest order. Is this acceptable for a production extension, or should a build system be added?

7. **Rate limiting**: The current rate limiter is in-memory (`RateLimitMiddleware` + `rate_limiter.py`). Should it be backed by Redis (config is already in `Settings.redis`) for multi-worker support?

8. **Alembic migrations**: The project has Alembic configured (`backend/migrations/`) but `container.py` calls `create_all_tables()` directly. Should Alembic be the sole migration mechanism?

9. **`analytics.py` OCR in route handler**: The `_collect_datasets_files()` function calls `container.solver_service.submit()` to OCR failed captcha images. Is this acceptable, or should it be a background task?

10. **Module communication**: In a modular monolith, how should modules communicate? Direct service calls, events, or a message bus? What is the right level of coupling?
